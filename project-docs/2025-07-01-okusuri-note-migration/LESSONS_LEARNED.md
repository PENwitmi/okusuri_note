# okusuri_note 移行作業から学んだ教訓

作成日時: 2025-07-01 02:40  
作成者: CEO

## 概要
プロジェクト名移行作業を通じて得られた重要な教訓をまとめ、将来の参考とします。

## 1. 表記揺れの影響

### 発見した問題
- **pharma_dex** vs **pharm_dex**: 2つの表記が混在
- **Drug Base** vs **PharmaDx**: ブランド名の不統一
- **影響**: 開発者の混乱、検索の困難、保守性の低下

### 学び
**プロジェクト初期に命名規則を確立することの重要性**
- 技術名とブランド名を明確に分離
- すべての開発者に周知徹底
- ドキュメント化して参照可能に

## 2. 作業範囲の明確化

### 発生した問題
- 「全ファイルで更新」という曖昧な指示
- 開発者が_old_files/や使用していないツールまで確認

### 改善策
```
## 更新対象（明確に）
- docs/内の公開ファイルのみ
- 現在使用中の設定ファイル
- README.md、CLAUDE.md

## 更新対象外（明確に）
- _old_files/内のすべて
- backup_*ディレクトリ
- 使用していないツール
```

## 3. Git作業の前準備

### 良かった点
- CEOがディレクトリ名とGit remote URLを先に変更
- 開発者の混乱を防いだ

### 推奨フロー
1. 基盤的な変更はCEOが先行実施
2. 明確な状態で開発者に作業依頼
3. 完了後にCEOがGit操作

## 4. 並列実行の効果

### 成果
- 3名の開発者が同時作業
- 予定45分 → 実績12分（73%短縮）

### 成功要因
- 明確な役割分担
- 依存関係の少ないタスク設計
- 適切な優先順位付け

## 5. 品質確認の重要性

### 実施内容
- Dev1による最終品質チェック
- 主要ページ表示、リンク動作、命名規則統一性

### 効果
- エラーゼロでの完了
- ユーザー影響なし
- 安心してGitプッシュ可能

## 6. ドキュメント管理

### 反省点
- 移行計画書をproject-docs/直下に配置

